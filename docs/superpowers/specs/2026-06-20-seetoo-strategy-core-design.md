## 1. 定位

### 1.1 兩個角色
- **被動底座**：餵三支技能。它們的 SKILL.md 會點名讀取核心 `references/` 的特定檔案。
- **可直接查**：使用者亦可直接叫核心做「忠實查詢」（例：「司徒達賢對垂直整合
  怎麼說？」→ 原文摘錄 + 章節出處）。

### 1.2 是什麼 / 不是什麼
- **是**：從講義蒸餾出的、可引用且可執行的知識庫。
- **不是**：不是把 PDF 整本搬入；不是一般 MBA 策略百科。只裝講義內容，
  外部知識一律標記為〔延伸·非講義〕。

### 1.3 三條保證
1. **術語鎖**：輸出鎖死司徒達賢的用詞，不得替換為一般 MBA 術語。
2. **原文錨定**：每個操作主張都必須錨回某條原文依據；錨不回 ⇒ 標〔待查〕或不寫。
3. **反主張邊界**：守住第十一章「不從 mission/vision、環境分析、SWOT、設定目標
   起手」等硬邊界，任何技能輸出踩線即自我攔截。

## 4. 雙層模板（每份 reference 檔的固定格式）

每份主題檔固定兩層，並標來源。範例（原文取自已讀之「重點摘要」p1）：

```markdown
# 策略構想　〔出處：重點摘要；第三章〕

## 原文依據
- 〔原文〕「策略構想是指一套策略之所以可行，究竟是如何配合了外界的環境？
  如何配合了本身的條件？如何配合了各方的期待（目標組合）？」（重點摘要,p1）
- 〔摘述〕過去成功策略中這些「配合」是既存事實；未來策略中則是
  「有待驗證的前提假設」。（重點摘要,p1）
- 術語鎖：策略構想／前提假設／環條目 — 不可替換為「商業模式」「value proposition」

## 操作層
- [O1] 推導策略構想時逐一問：如何配合 環境？條件？目標組合？六大構面如何呼應？ ⟵錨:原文1
- [O2] 每個「配合」寫成一條可檢驗前提，標記「既存事實／待驗證」 ⟵錨:原文2
```

### 三種標籤
- **〔原文〕** 逐字引述；必附章節 + 頁碼。摘錄限關鍵定義與招牌語句，非整段照搬。
- **〔摘述〕** 忠實轉述；必附章節 + 頁碼。
- **〔延伸·非講義〕** 外部補充；極少用，須明標非講義，且不得與原文依據衝突。

## 5. 紀律機制（反杜撰硬規則，寫進 SKILL.md）

1. **操作層每條必須 ⟵ 錨回某條原文依據編號**。錨不回 ⇒ 標〔待查〕或不寫。
2. **術語鎖**：輸出時不得把講義術語換成一般 MBA 詞（清單見 `00-術語表.md`）。
3. **反主張攔截**：任何技能若想從 SWOT/mission/環境分析/設定目標 起手，撞
   `09-反主張與方法比較.md` 的邊界並自我攔截、改回形態分析法路徑。
4. **可回溯**：每份檔在 `章節索引.md` 標來源 PDF 檔/頁，可一鍵回對照。

---

## 6. 消費契約（三支技能如何讀核心）

- 核心對外暴露**穩定的檔名**作為「API」；三支技能在自己的 SKILL.md 內以
  相對路徑點名讀取，例如 `references/05-十步驟程序.md`。
- 核心的 `SKILL.md` 提供「何時讀哪份」的對照表，作為三支技能的索引。
- 預期消費對應（建置三支技能時細化，此處僅為設計意圖）：
  - ① 分析引擎：05 為主程序，01/02/03/04/06 為框架，07/08 為道理透鏡。
  - ② 拷問教練：02（前提驗證）、09（反主張）、01（六大構面）為追問彈藥。
  - ③ 學習助教：全部；另搭配 `sources/` 之例題/習題簡報（建置期決定如何納入）。

## 7. 驗證方式

- **可回溯稽核**：抽查操作層每條都有有效原文錨點（編號存在且對應正確）。
- **黃金查詢測試**：固定數題（垂直整合？策略構想？為何不從 SWOT 起手？）核心
  須以原文 + 出處作答，且用詞鎖定。
- **反杜撰測試**：問講義未涵蓋者（例：「他怎麼看 OKR？」）核心須答「講義未涵蓋／
  僅能標〔延伸·非講義〕」，不得編造。

## 8. 建置流程（供 writing-plans 細化）

1. 建立目錄骨架與 `SKILL.md`、`00-術語表.md`、`章節索引.md`、`版本.md`。
2. 逐章精讀 PDF → 套雙層模板填寫 → 標頁碼。**各章可平行**（一章一工作單元）。
3. 每份檔完成後跑「可回溯稽核」。
4. 全部完成後跑「黃金查詢 + 反杜撰」測試。
5. 部署：建立 symlink 至 `~/.claude/skills/`，驗證 `/seetoo-strategy-core` 可叫用。

## 9. 非目標（YAGNI）

- 不處理英文版／多語；輸出語言為繁體中文。
